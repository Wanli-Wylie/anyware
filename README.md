# Anyware

**从代码库中挖掘可测试结构，赋能经过实证验证的重构与迁移。**

> [!NOTE]
> 本仓库是一个持续更新的工作空间，将不断补充技术白皮书、实验结果、生成的工件及设计文档。

## 1. AI 时刻

AI 领域的领军人物正在形成一个共识：软件即将发生根本性变革。

**Andrej Karpathy**（OpenAI 联合创始人、前特斯拉 AI 负责人）表示，这是"二十年来我编程工作流最大的变化"。他描述了一个新时代——*智能体工程*（agentic engineering）——开发者不再逐行编写代码，而是指挥 AI 智能体来编写、测试和重构代码。

**Dario Amodei**（Anthropic CEO）预测，到 2026 年底，AI 将有能力完成软件工程师"大部分、甚至全部"的编码工作。Anthropic 内部已有工程师表示他们不再直接编写代码。

**黄仁勋**（NVIDIA CEO）提出了不同观点：AI 不会取代工程师，而是让他们的生产力大幅提升。职业的形态会改变，但规模不会缩小。

### COBOL 地震

2026 年 2 月 23 日，Anthropic 发表了一篇博客——"[AI 如何打破 COBOL 现代化的成本壁垒](https://www.anthropic.com/research/cobol-modernization)"——声称 Claude Code 能够跨数千行 COBOL 代码映射依赖关系、记录无人记得的工作流，并在"几个季度而非几年内"完成代码库现代化。

市场反应立竿见影：**IBM 单日市值蒸发超过 310 亿美元**——创下 2000 年以来最大跌幅。埃森哲和 Cognizant 的股价同样下挫。一篇博客就撼动了整个遗留系统现代化行业。

各大科技巨头正在全力争夺这一市场：
- **AWS Transform** —— 将 COBOL 和 PL/1 "在数分钟内"转换为 Java 的智能体式 AI 服务
- **微软 CAMF** —— 用于 COBOL 到 Java 迁移的开源多智能体框架，已在 Bankdata（超过 7000 万行大型机代码，服务丹麦 30% 的银行业务）上测试
- **Google Cloud** —— 基于 Gemini 的大型机代码改写服务，与 Kyndryl 合作
- **IBM Project Bob** —— 多模型 AI IDE，集成 Claude、Llama、Mistral 和 Granite，IBM 内部 3,000 名开发者每日使用

遗留系统现代化市场在 2025 年已达到 **250 亿美元**，预计 2030 年将增长至 **570 亿美元**。Gartner 预测，到 2026 年 40% 的遗留系统现代化项目将采用 AI，而 2023 年这一比例不到 10%。

他们在一件事上达成一致：**AI 将编写新代码。** 但现有的代码怎么办？

全球有数万亿行遗留软件正在运行——气候模型、金融系统、国防基础设施、科学仪器。这些代码库历经数十年构建，承载着不可替代的领域知识。它们不能被丢弃，必须**迁移**到现代语言和平台。

而迁移需要当前 AI 系统尚无法提供的东西：**新代码与旧代码行为完全一致的证明。**

## 2. 问题：重构与迁移

遗留代码库陷入了困境。它们所使用的编程语言无法利用现代硬件（GPU、云计算、并行处理器）。原始开发者正在退休。它们依赖的工具已不再维护。

目前有两种方法——但都不完整：

| 方法 | 能做什么 | 不能做什么 |
|------|---------|-----------|
| **AI 翻译**（如 GPT、Claude） | 生成目标语言的代码 | 证明翻译是正确的 |
| **代码分析工具** | 发现现有代码的属性和模式 | 将代码转换为新语言 |

AI 能*翻译*但不能*验证*。分析工具能*验证*但不能*翻译*。真正的挑战是**同时做到两者**。

### 行业已经意识到了这一点

IBM 立即反驳了 Anthropic 的说法：*"翻译几乎无法捕获真正的复杂性。数十年的软硬件集成不是简单搬运代码就能复制的。"*

代码分析公司 Phase Change Software 警告说，仅靠生成式 AI 的代码翻译准确率只有 **74%**——*"对于合规要求、现代化规范和事故复盘来说，26% 的错误率是一场等待发生的灾难。"*

Futurum Group 分析师给出了最尖锐的评价：**"无论是 IBM 还是 Anthropic，都未能在受监管行业中展示遗留代码与迁移代码之间可证明的行为等价性和生产部署证据。"**

可证明的行为等价性。这就是缺失的那块拼图。

### 更深层的问题

要证明翻译是正确的，你需要用相同的输入运行新旧两个版本，然后检查它们是否产生相同的输出。这只有在每段代码能够**独立测试**——被提取出来，独立运行并检查——时才有效。

大多数遗留代码做不到这一点。函数读取隐藏的共享数据，写入文件，依赖网络通信，或以各种方式与系统其余部分纠缠在一起，无法进行隔离测试。

**真正的瓶颈不是遗留代码太复杂而难以翻译，而是遗留代码太纠缠而难以测试——没有测试，任何翻译都不可信。**

## 3. 我们的洞察：可测试性是核心

### 一个日常类比

想象你继承了一座拥有 3,000 台机器的工厂。这些机器能运转，但已经老旧——速度慢，维护成本高，零件也买不到了。你想用现代设备逐一替换。

怎么知道替换件好不好用？很简单：把相同的原料分别送入新旧机器，检查它们是否生产出一样的产品。如果在大量输入下输出都一致，替换就通过了验证。

但问题来了。这座工厂里大多数机器不是独立的。它们与相邻机器共用管道，从一个几十台机器共享的中央控制面板读取数据，或者会触发全厂报警。你没法把一台机器搬到测试台上，因为它**和其他一切都连在一起**。

于是你开始研究这些连线。然后你发现了一件令人惊讶的事：大多数连线是死的。这台机器读取的控制面板数值？多年来没人改过——实际上是个常量。它触发的报警？只是通过一个监控包装器触发的，测试时可以绕过。共用的管道？只有单向流动。

**机器看起来是纠缠的，但它的行为是独立的。** 去掉那些死连线，它就能上测试台了。现在你可以验证它的替换件了。

这正是遗留软件的真实处境——也是我们的核心洞察。

### 一种不同的思维方式

软件行业把可测试性当作锦上添花——一种有空才投入的质量实践。我们的看法不同。

**可测试性是基础。** 它是让一切成为可能的属性。没有它，重构是猜测，翻译是祈祷，迁移是表演。有了它，每次修改都可以检查，每次翻译都可以验证，每次迁移都带有证明。

一个函数能否独立运行、接受确定的输入、始终产生相同的输出？如果能，它就是可测试的——它的任何替代版本都可以被机械地验证。如果不能，你就是在盲目操作。

### 大多数障碍是偶然的

这是我们的核心发现：**阻碍测试的因素大多是偶然的，不是根本性的。**

遗留代码*看起来*深度纠缠，但大部分纠缠是表面的。通过系统性分析，我们发现：

- 声明为可变的变量，**实际上从来没人改过**——它们其实是常量
- 看起来依赖共享数据的函数，**实际上只是读取**——它们的行为就像纯计算
- 标记为"并行"的函数，**只是通过日志包装器才接触到并行代码**——核心计算是独立的

在真实的科学计算代码库中，**60–80% 看起来像危险共享状态的数据实际上是只读的。**

| 代码库 | 实际只读 | 单写入者配置 | 真正共享 | 存在别名 |
|--------|---------|-------------|---------|---------|
| gfdl-radopt（357 个函数） | 96.2% | 1.9% | 1.9% | 0% |
| CICE（890 个函数） | 43.0% | 15.3% | 25.4% | 0.1% |
| NEMO（3,804 个函数） | 62.5% | 16.7% | 13.5% | 0.2% |

可测试的结构已经存在，只是被掩埋了。我们的工作是**挖掘**——系统性地发掘表面复杂性之下已经干净的部分。

### AI 智能体可以挖掘可测试子结构

AI 智能体在这里发挥的作用不是代码翻译，而是**可测试结构的矿工。**

我们提供地图：一份系统性的目录，标明哪些因素阻碍了可测试性，代码的声明与实际行为之间存在哪些差距。AI 智能体沿着地图工作——机械地清理不必要的声明，将虚假的共享数据转换为显式输入，移除无用的纠缠。每一轮处理后，更多代码变得可以独立测试。循环往复。

以 NEMO（一个包含 3,804 个函数的海洋建模代码库）为例：

| 挖掘阶段 | 可独立测试的函数比例 |
|---------|-------------------|
| 分析前 | 31.9% |
| 完整分析后 | 55.6% |

这意味着**额外 904 个函数**变得可以独立验证——而代码的计算逻辑没有任何改变。每一个都成为经验证 AI 翻译的候选对象。

### 为什么这很重要

传统方法构建的是各自独立的工具——这里一个代码检查器，那里一个测试框架，另一边一个 AI 翻译器。它们彼此不协作。

从可测试性作为基础出发，一切统一起来：

- **重构** = 扩大可测试表面（挖掘）
- **翻译** = AI 为可测试的部分提出等效代码（创造性搜索）
- **验证** = 运行新旧两个版本并比较输出（实证证明）
- **迁移** = 将经过验证的部分组装成新的代码库（组合）

每一步为下一步提供输入。每一步都可以独立检查。系统之所以能组合，是因为基础——可测试性——本身就是可组合的。

## 4. 系统架构

两个产品，一个共享数据层。

**产品一——重构**：接收一个代码库，在不改变代码行为的前提下，使更多部分变得可测试。它分析可测试性障碍，识别代码声明与实际行为之间的差距，并按优先级执行安全改写。每一轮循环都扩大可测试表面。即使不进行迁移，这本身就有价值。

**产品二——迁移**：接收可测试的函数，产出经过验证的翻译。对每个函数，系统生成测试输入，捕获原始版本的输出，让 AI 提出翻译方案，然后用相同的输入运行翻译版本并比较结果。每个翻译后的函数都附带一份机器可读的证书：测试了什么、期望的结果、实际的结果、是否匹配。

**共享数据层**——各阶段之间的所有通信都通过结构化数据库进行（13 个模式，46 张表）。添加新的编程语言意味着向现有表中添加新行，而不是重新设计系统。该架构已在五个生产代码库上验证，规模从 40 到 3,126 个源文件（最多 15,658 个函数），零数据丢失。

```
┌──────────────────────────┐     ┌──────────────────────────┐
│  产品一：重构              │     │  产品二：迁移              │
│  扩大可测试表面            │ ──▶ │  经验证的 AI 翻译          │
│ （同一语言内）             │     │ （跨语言）                │
└────────────┬─────────────┘     └────────────┬─────────────┘
             │                                 │
             └────────────┐  ┌─────────────────┘
                          ▼  ▼
              ┌──────────────────────┐
              │  共享数据库层          │
              │  13 个模式 · 46 张表   │
              │  语言无关              │
              └──────────────────────┘
```

### 超越迁移：持续性能优化

迁移回答的问题是：*新代码是否和旧代码做同样的事？* 但自然会有下一个问题：*新代码能否做得更快？*

Google DeepMind 的 **[AlphaEvolve](https://deepmind.google/blog/alphaevolve-a-gemini-powered-coding-agent-for-designing-advanced-algorithms/)**（2025）展示了一个强大的范式：将提出代码变更的 AI 与衡量结果的确定性评估器配对，然后进化出最优候选方案。通过这个循环，AlphaEvolve 发现了超越 50 年历史数学结果的算法，在关键 AI 训练操作上实现了高达 32.5% 的加速，并通过更优的调度策略持续回收了 Google 全球 0.7% 的计算资源——已在生产环境中运行超过一年。

这个模式是：**提出 → 评估 → 选择 → 重复。**

Anyware 的可测试性基础设施恰好为这个循环创造了前提条件。一旦函数可测试并已通过验证的测试套件完成迁移，同一套基础设施就能支持持续的性能优化：

1. AI 提出已翻译函数的更快实现方案
2. 现有测试套件验证正确性（相同输入，相同输出）
3. 基准测试衡量性能提升
4. 选择最优候选方案，循环往复

可测试性不仅支撑迁移——它还支撑一个**持续改进引擎**，其中每一次优化都经过实证验证。证明"这个翻译是正确的"的同一张证书，也能证明"这个优化是正确的，*而且*快了两倍"。

## 5. 与 OpenClaw 的连接

[OpenClaw](https://github.com/openclaw/openclaw) 是增长最快的开源 AI 智能体框架——一个 AI 直接操作代码库的系统：读取代码、提出修改、执行命令、迭代改进。它代表了一类工具（还包括 Claude Code、Cursor、Windsurf），正在成为软件工程的双手和眼睛。

这些智能体是**执行层**——它们能够阅读遗留代码、提出重构方案、生成翻译、运行测试。

它们缺少的是**验证层**——一个结构化的系统，告诉它们：
- 哪些函数已准备好被翻译
- 哪些障碍阻止了测试
- 在哪里挖掘可测试结构
- 如何验证翻译是否正确
- 是否可以信任结果

Anyware 提供这个缺失的层。

```
┌──────────────────────────────────────────────────┐
│  AI 智能体（OpenClaw / Claude Code / Cursor）      │
│  提出重构方案和翻译                                 │
│  基于测试失败进行迭代                               │
└───────────────────────┬──────────────────────────┘
                        │
                   读取和写入
                        │
┌───────────────────────▼──────────────────────────┐
│  Anyware                                          │
│  分析可测试性，运行验证，                            │
│  记录证据链，扩展可测试边界                          │
└──────────────────────────────────────────────────┘
```

智能体负责**创造性工作**——提出基于规则的系统无法生成的代码变更。流水线负责**证明**——检查每次变更是否保持了正确性，记录证据，并扩展可测试的范围。

更换智能体、更换 AI 模型、更换目标语言——验证层始终不变。

**我们的判断：能够挖掘可测试结构并证明翻译正确的 AI 智能体，将胜过那些只是生成代码然后祈祷的智能体。**
