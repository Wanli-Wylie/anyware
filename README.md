## 1. 引论

### 1.1 软件即抽象

程序将现实世界中的过程——物理模拟、金融清算、控制回路——编码为计算机可执行的形式。编码必然伴随抽象：选取过程中值得关注的侧面，以数据和操作表达之，其余则舍弃。气候模型将大气表达为网格场与有限差分算子，而非单个分子。交易系统将市场表达为订单簿与撮合引擎，而非每位参与者的决策。抽象使编码成为可能——它是软件应对世界复杂性的根本手段。

软件承载可复用的逻辑并由抽象管理，因此它必然要演化。它所建模的世界在变——物理学不断精化，法规持续更替，需求日益增长。它所运行的硬件在变——新架构、新加速器、新存储层次。它所栖息的生态在变——语言老化、团队更迭、平台迁移。维护、优化、迁移并非工程上的麻烦，而是软件本质的必然结果：被捕获的逻辑，注定比诞生它的语境更长寿。

然而，我们用来表示软件的载体——源文件、语法树、编译单元——本是为另一目的设计的：供编译器读取。它们编码了程序**说了什么**，而非程序**意味着什么**。两个用不同语言实现相同函数的程序，在这种表示下不具有任何形式关联。对一个函数的局部修改，可能经由表示未曾显式呈现的共享状态，悄然破坏另一个函数。这些表示服务于编译，而非演化。要找到更好的表示——一种服务于跨时间复用的表示——我们必须追问：抽象背后，究竟什么是不变的？

### 1.2 不变量存在

每一层软件抽象背后，都存在不随编码改变而改变的东西。

一个气候模型从 Fortran 重写为 Python，程序全然不同——语言、语法、内存布局、调用约定都变了。然而我们一眼就能认出它是**同一个模型**。运动方程相同，热力学关系相同，数值格式对相同输入产生相同结果。这些计算既不属于 Fortran 版本，也不属于 Python 版本——它们属于被建模的物理过程本身。海水密度作为温度与盐度的函数，不因其由 Fortran 的 do 循环、NumPy 的广播运算还是手工计算而改变，也不因机器是 1985 年的 Cray 还是 2025 年的 GPU 而改变。这一计算之所以不变，是因为它是关于世界的事实，而非关于某种编码的事实。

程序还与资源交互——文件、网络、数据库、设备。交互方式因平台而异：Fortran 与 Python 读文件的调用完全不同。但其模式并不改变：每一次资源交互都遵循寻址、观察或修改的范式。这一模式之所以不变，是因为它反映的是资源本身的性质，而非某种特定接口。

程序还有组织——先后次序、条件分支、单元间的调用关系。重写程序时，组织结构在各部分被重新表达的过程中得以存续。拓扑结构之所以不变，是因为它反映的是过程的逻辑依赖，而非任何语言的语法。

三个不变量存在于每一层软件抽象背后：程序所执行的**计算**、程序所触及的**资源**，以及将两者组织起来的**结构**。它们不是程序的性质，而是程序所编码的过程的性质。只要软件的**行为**得到保持，这些不变量就持续存在。

### 1.3 向形式世界的投影

不变量是关于现实过程的事实。要对其进行机械化推理——验证、变换、组合——它们必须被投影为形式对象。

**计算**投影为*纯函数*：从输入到输出的确定性映射，不依赖任何外部状态。纯函数是不变计算的形式对应物。其行为是数学事实：相同输入，相同输出，永远如此。两个纯函数等价，当且仅当它们在所有输入上一致——这一性质可通过采样进行经验检验。

**资源**投影为*状态接口*：(索引, 命令, 查询) 三元组。索引标识资源，命令修改资源，查询观察资源。这一三元组是所有资源交互背后不变模式的形式对应物。文件、数据库、网络套接字、共享变量，无论平台与语言，都可归约到这三种操作。

**组织**投影为*结构*：一个有向图，其节点是纯函数与状态接口，其边为顺序、分支或调用-返回。该图是不变拓扑的形式对应物，使程序架构显式化、可机器导航。

三元模型即此投影：

| 现实不变量 | 形式对象 | 验证方式 |
|-----------|---------|---------|
| 计算 | 纯函数 | 差分测试（相同输入 → 相同输出） |
| 资源交互 | 状态接口（索引, 命令, 查询） | 结构同一性 |
| 组织 | 有向图 | 拓扑保持 |

每个形式对象是一个开放的插槽，可附着各种属性：性能、精度、一致性、延迟、数值稳定性。属性充实模型但不改变其结构——骨架是三个不变量，属性则视具体分析而定。

本文展开三元模型的形式化发展，并论证它为自主智能体持续演化软件提供了充分的程序表示。

### 1.4 大语言模型的角色

三元模型定义了不变量**是什么**以及它们**如何**投影为形式对象。但从现有软件中提取不变量——将纯计算与状态交互和组织结构分离——需要理解代码的**语义**。物理模块中名为 `dt` 的变量是时间步长，被数百个函数读取，却无人写入。`MPI_Wtime` 的调用是基础设施，不是算法。诊断性的 `write` 是日志，不是计算。这些都是语义判断，需要阅读代码并理解其意图。

大语言模型能做出这些判断。它能阅读一个函数，分辨哪些是计算、哪些是环境交互；判断一个模块变量从未被写入因而是常量；识别一个调用属于基础设施而可以移除且不影响结果。它能**定位不变量**——从缠结的整体中提取纯计算——因为它理解代码的语义内容。

但它无法**验证**提取的正确性。大语言模型可以判定一个函数为纯函数，但无法证明。三元模型弥合了这一鸿沟：一旦不变量被提取并投影为形式对象，机械化验证即可施行。纯函数通过输入采样来测试，状态接口通过结构同一性来检查，图通过拓扑来比较。大语言模型负责提取，形式系统负责验证。两者单独都不够，合在一起，则使软件能够在每一步都附带机器可检查的证据持续演化。

### 1.5 面向智能体的表示

既然不变量存在于软件背后，大语言模型能定位它们，形式系统能验证提取的正确性——那么一种新的软件表示便成为可能。在这种表示中，程序不以"恰好能编译的文本"存储，而以经过分解的结构存储：已验证的纯函数、类型化的状态接口、显式的图。一种不是为编译器而是为智能体设计的表示。

在这样的表示中，每个纯函数携带其正确性的证据——用于验证它的输入分布与输出比较。每个状态接口是可机械检查的类型化契约。每条结构边是智能体可以导航、重连或优化的显式依赖。该表示不是在分析过程中临时访问后随即丢弃的中间产物，而是软件的持久存在形式。

这改变了可能性的边界。维护变为局部操作：修改一个纯函数，对其在输入分布上重新验证，即告完成。优化变为增量操作：用更快的实现替换一个纯函数，证据协议不变。迁移变为组合操作：逐一翻译各纯函数，逐一验证，整体的正确性由各部分推出。软件的三项固有需求——维护、优化、迁移——成为在一种专为支持它们而设计的表示上的常规操作。

本文其余部分正式展开三元模型。第 2 节定义形式对象：纯计算、状态操作与结构。第 3 节介绍 SoftwareMiner——从现有代码库中提取这些对象的神经-符号智能体。第 4 节讨论该模型在数值模拟领域的应用影响。

## 2. 形式对象

### 2.1 纯计算

纯计算是一个代码片段——无论是数学规约、子程序还是任意代码段——仅依赖于其显式输入并产生确定性输出。它是现实中不变计算向形式世界的投影。

实际中，纯计算很少以孤立形式出现。一个子程序会读取模块变量、继承被调用者的副作用、写入诊断输出。计算确实存在，但它与其环境纠缠在一起。要获取形式对象，必须对计算进行**局部化**：将其外部依赖解析为显式参数，使之自包含。

**局部化。** 给定一个含有外部引用的代码片段，局部化通过以下方式生成自包含版本：

- 将只读模块变量提升为显式参数。
- 将种类参数、类型定义和常量提取到局部规约块中。
- 将环境交互（I/O、通信）分离为独立的状态接口。

结果是一对：

```
spec  = 局部声明（种类、类型、常量）
exec  = 可执行代码（USE 语句已移除，引用已解析）
```

局部化形式 `spec ∥ exec` 即为形式对象。它可独立编译，接受输入，产生输出，可被测试。

**局部化以规范形式为目标。** 在给定的规范化策略下——哪些引用提升、哪些提取、哪些分离——局部化的目标是生成所有输入显式化、所有环境交互移除的自包含形式。原理上这是一个机械变换：根据引用的观测生命周期将其改写为参数。实践中，机械化版本需要完整的数据流分析、别名解析和跨模块生命周期追踪。我们用大语言模型来近似完成：模型阅读代码及其上下文，识别哪些引用是只读常量、哪些是配置项、哪些是真正的共享状态，并生成候选的局部化形式。然后由形式系统**验证**结果：符号检查（作用域逃逸扫描）确认局部化形式中没有变量越出其显式接口。若检查返回零障碍，则局部化获得认证。验证标准是确定性的；生产过程不是。

**障碍。** 符号检查探测以下类别的作用域逃逸：

| 类别 | 示例 |
|------|------|
| 共享内存 | COMMON 块、SAVE 变量、初始化导致的隐式 SAVE |
| 外部 I/O | 文件读写、标准输出、OPEN/CLOSE |
| 通信 | MPI 调用、协数组操作、OpenMP 指令 |
| 未解析引用 | 对未知函数的调用、继承的种类参数、外部类型定义 |
| 不可序列化接口 | 派生类型参数、指针参数、假定大小数组 |

每个障碍都是关于代码的具体可枚举事实，而非启发式判断。该检查是**保守的**：零障碍意味着机械可测试性——代码可独立编译、接受生成的输入、在所采用的检查模型下产生确定性输出。对于实际上语义纯粹、但因其纯粹性依赖于局部分析不可见的不变量（别名、隐藏的非确定性、浮点环境依赖）而被报告障碍的代码，这种不完备是无害的——可通过进一步局部化来解决，而非削弱检查标准。

### 2.2 状态

状态接口是资源交互的形式投影。无论资源类型如何，每一次资源交互都可分解为三个原语：**索引**（标识资源）、**命令**（修改资源）、**查询**（观察资源）。

（索引, 命令, 查询）三元组刻画了不可约的交互模式。它并不捕获有状态交互的全部语义：并发控制、事务保证、一致性模型、顺序约束、故障模式、时序行为在真实系统中都很重要。这些是按需附着于状态接口之上的**属性**——三元组是结构骨架，不是完整描述。

但资源并非千篇一律。它们沿一条光谱分布，由资源所提供的保护与组织程度所定义：

**自管理资源**拥有独立的生命周期、身份和访问协议。数据库管理自身存储、施行模式约束、处理并发访问，并在进程终止后存续。Docker 容器封装了完整的执行环境。这些资源通过名称或句柄寻址，其命令/查询接口是定义良好的 API。状态接口构成清晰的契约：程序发出命令与查询，资源管理其余一切。

```
数据库    : (连接 + 表 + 键,      {insert, update, delete},  {select, count})
容器      : (容器 id,             {start, stop, exec},       {inspect, logs})
```

**半组织资源**具有一定的结构和协议，但隔离性较弱。文件有路径和格式，但无模式强制——程序必须自行解读其内容。网络套接字有地址和传输协议，但除协议本身提供的保证外，不保证投递或顺序。这些资源提供更薄的契约：程序须做更多工作才能正确使用。

```
文件系统  : (路径或单元号,        {open, write, close},      {read, inquire})
网络      : (地址 + 端口,         {send, connect, close},    {recv, accept})
MPI       : (通信子 + 进程号,     {send, bcast, scatter},    {recv, allreduce})
```

**无保护资源**没有生命周期管理、没有访问协议、没有隔离。裸内存——模块级变量、COMMON 块、SAVE 变量——是极端情形。任何能看到名字的代码都可以读写它。没有契约，只有约定。这是遗留代码中缠结的主要根源，同时也是最容易解决的资源类型：经验表明，60%–80% 的模块变量从未被写入，它们实际上是常量而非真正的状态。（以上数据来自对 NEMO、CICE 和 GFDL 辐射代码库的试点分析；更广泛的验证正在进行中。）

```
模块作用域   : (限定名,               {赋值},        {引用})
COMMON 块    : (块名,                 {写入成员},    {读取成员})
SAVE 单元    : ((子程序, 变量),       {赋值},        {引用})
```

光谱位置之所以重要，在于它决定了状态接口的**生命周期**，从而决定了局部化过程中的处理方式：

| 生命周期 | 典型资源 | 命令集 | 处理方式 |
|---------|---------|--------|---------|
| 只读 | 模块常量 | ∅ | 消除——提升为参数 |
| 配置 | 初始化一次的变量 | {init} 一次 | 提升为参数 |
| 累加器 | 累加和、计数器 | {accumulate} | 提升为 in-out 参数 |
| 共享可变 | 真正的共享状态 | {assign} | 外化为参数 |
| 托管 | 数据库、容器 | 完整 API | 保留为类型化桩 |

在光谱的一端，"状态"其实并非状态——它是伪装为变量的常量。在另一端，状态是拥有独立生命周期的真正外部资源。局部化过程消除前者，将后者类型化为显式接口。

### 2.3 结构

结构是将纯计算与状态接口组合为程序的图。它是不变组织向形式世界的投影。

结构是分层的。基本组织单元是**作用域**——一个具有明确边界的命名代码区域：

| 作用域类型 | 边界 | 包含内容 |
|-----------|------|---------|
| 模块 | `module` ... `end module` | 类型、常量、接口、所含子程序 |
| 子程序 | `subroutine`/`function` ... `end` | 声明、可执行语句 |
| 主程序 | `program` ... `end program` | 顶层执行 |
| 数据块 | `block data` ... `end block data` | COMMON 块初始化 |

在每个作用域内部，语句有序排列为三个部分：

```
环境    :  USE, IMPORT, IMPLICIT    — 本作用域依赖什么
声明    :  类型、参数、变量          — 本作用域定义什么
执行    :  赋值、调用、循环          — 本作用域做什么
```

这种内部三分结构与三元模型本身形成镜像：环境部分声明状态依赖，声明部分定义纯计算所需的数据，执行部分承载计算与控制流。

**作用域嵌套定义层次结构。** 模块包含子程序，子程序可包含内部子程序。嵌套是严格的：每个作用域至多有一个父作用域，子作用域完全包含在父作用域的边界内。该层次结构即为结构图的骨架。

**边连接作用域。** 作用域之间存在三类边：

- **顺序**：同一作用域内一个块在另一个块之后执行。
- **分支**：条件或迭代选择（IF、DO、SELECT）。
- **调用-返回**：调用另一个作用域，控制权转移后再返回。

结构图 *G* = (*V*, *E*) 的节点为纯计算、状态接口或调用-返回链接，边为顺序、分支或调用-返回。它是一个有向图，使程序的架构显式化、可由机器导航。

**在现有模型中，并发通过状态操作与图组合来表示**，而非引入第四种原语。通道、共享内存与同步原语都是状态资源——通过索引寻址、由命令修改、由查询观察。其组合在结构图中表达。并发推理的复杂性来源于所产生的状态空间规模，而非需要额外的结构范畴。

### 2.4 等价性验证

两个纯计算**等价**，当且仅当它们对相同输入产生相同输出。建立此等价性容许多个严格度层次：

**第一层：经验性——差分测试。** 从输入模式上的分布采样输入，分别执行两个实现，在容差范围内比较输出。这是最实用的层次：只需两个版本均可编译并运行。

证据质量取决于输入分布的质量。智能体可以*迭代地探索*输入空间：从参数声明范围上的均匀分布出发，观察哪些区域产生分歧输出或数值不稳定，然后精化分布使之集中于边界情形、退化输入和极端值。证据是统计性的——它不证明等价性，但建立了受分布覆盖度约束的信心。

输入分布本身也可以推断。给定一个纯函数的签名——参数名、类型、秩、维度——智能体可据此推理物理约束（温度为正、密度有界、网格索引为数组范围内的整数），并生成尊重这些约束的分布。推断是迭代的：智能体提出方案、测试、观察失败、精化。

**第二层：符号性——形式验证。** 对于可进行符号分析的纯计算——有界循环、有界域上的算术运算、无浮点特殊情形——符号执行或 SMT 求解器可对所有输入（而非仅采样输入）建立等价性。证据更强，但适用范围更窄。

**第三层：混合。** 符号分析覆盖可处理的核心部分，差分测试覆盖其余部分。证据记录同时携带两者：已证明的区域与已测试的区域，并明确标注何处已验证、以何种方式验证。

对于状态接口，等价性是结构性的：两个接口等价，当且仅当它们具有相同的（索引, 命令, 查询）签名。这是语法检查——精确且可机械执行。

对于结构，等价性是拓扑性的：两个图等价，当且仅当它们具有相同的节点（模纯计算等价性和状态接口同一性）和相同的边。这是图比较——精确且可机械执行。

在三元分解下——纯计算与状态完全隔离，状态接口显式类型化——整体程序等价性归约为上述三项独立检查。这一组合性质在分解边界得到维护的条件下成立：纯块不共享隐藏状态，状态接口忠实地类型化，结构边捕获所有控制依赖。实践中，涌现性交互（顺序、时序、故障传播）可能需要在集成边界进行额外验证。分解是一种组合式设计规范，而非自动保证。

## 3. SoftwareMiner：一个神经-符号智能体

### 3.1 核心承诺

整个系统建立在一项保证之上：

```
inspect(code) = 0 障碍  ⟹  代码可测试
```

可测试意味着：代码可独立编译、接受生成的输入、产生确定性输出。从可测试性出发，一切皆可推演。可测试的函数可以与其翻译版本进行差分测试——因此它**可迁译**。可测试的函数可以与优化版本进行差分测试——因此它**可演化**。可测试的函数可以连同正确性预言机一起交给 AlphaEvolve——因此它**可优化**。

这一承诺不是"此代码正确"，而是"此代码**可检查**"。一旦可检查，正确性便成为可通过运行测试来回答的经验问题。在可检查之前，正确性只能依赖人的判断、代码审查和希望。

`inspect` 是一个纯文本扫描器。它接收一段代码，扫描其中的作用域逃逸引用——来自外层作用域的变量、文件操作、通信调用、未解析依赖——然后返回障碍列表。无需上下文，无需调用图遍历，无需编译。一个函数，输入一段文本，输出一组障碍。确定性，快速，可按内容哈希缓存。

简洁是刻意为之。核心并不试图理解代码，不推理语义，不推断意图，不评判正确性。它只回答一个问题：这段代码中是否有名字越出了其显式接口？若无，则代码自包含。若自包含，则可测试。若可测试，则万物由此而生。

三个额外的核心函数完成整个协议：

```
signature(code)          → 输入输出模式
drivergen(code, schema)  → 测试驱动（编译、运行、捕获 I/O）
run(code, harness, input) → 输出
```

这四个函数——`inspect`、`signature`、`drivergen`、`run`——构成一条封闭的验证流水线。给定一段通过 `inspect` 的代码片段，该流水线可机械地提取其模式、生成测试驱动、在任意输入上执行并捕获输出。无需人工介入，无需语义理解。纯粹的机械协议。

这就是智能体的符号一半。

### 3.2 神经一半

核心能验证，但不能**生产**。给定一个缠结的函数——读取模块变量、写入诊断输出、从被调用者继承 MPI 调用——核心报告障碍，但不解决障碍。

解决障碍需要理解代码的**含义**。物理模块中名为 `dt` 的变量是时间步长——它是否被写入过？被谁？它是启动时设定一次的配置项，还是计算过程中被修改的真正共享状态？`log_timing` 的调用——是算法的一部分还是诊断性的？移除它会不会改变结果？这些都是语义判断，需要在上下文中阅读代码。

大语言模型做出这些判断。它阅读函数及其所在的代码库，识别哪些障碍是偶然的（声明为变量的只读常量、被误认为计算的诊断调用），哪些是本质的（真正的 I/O、真正的通信），然后提出一个偶然障碍已被解决的局部化版本。

大语言模型的方案不被信任，而是被**检查**。核心对提出的局部化方案运行 `inspect`。若零障碍：认证通过。若不通过：剩余障碍作为反馈返回，大语言模型再次提出方案。

这就是神经-符号循环：

```
repeat:
    barriers = inspect(current_code)
    if barriers = ∅: 认证通过——停止
    proposal = LLM(current_code, barriers, context)
    current_code = proposal
```

大语言模型近似完成的变换，原则上需要完整的数据流分析、别名解析和跨模块生命周期追踪才能机械地计算——这是一个相当复杂的符号程序分析问题。大语言模型通过阅读代码并**理解**它来绕过这一复杂性。核心随后验证结果是否满足可测试性标准，而不关心结果是如何产生的。

两个组件单独都不够。核心无大语言模型，能验证但不能生产。大语言模型无核心，能生产但不能验证。智能体是两者的组合。

### 3.3 搜索结构

挖掘代码库是一个具有特定结构的搜索问题，智能体利用这一结构。

**版本树。** 每个函数构成一棵树的根。每个节点是函数的一个版本。每条边是一次变换——**重写**（同一函数，更紧的接口）或**分解**（函数拆分为纯粹内核与命令式外壳）。当某个节点通过 `inspect` 且零障碍时，搜索终止。

**风险排序。** 变换按风险从低到高尝试：

1. **标注** ——收紧声明（零风险，高收益：在试点代码库中，60%–80% 的模块变量是只读的）
2. **提升** ——使隐式输入显式化（低风险）
3. **削除** ——移除非算法性调用（低风险，高收益：在 NEMO 中，约 90% 的 PARALLEL 分类是经由基础设施继承的）
4. **分解** ——将核心与外壳分离（中等风险，在重写不够时使用）
5. **展平** ——将不透明类型展开为原语（中等风险）
6. **内联** ——合并被调用者以打破继承的障碍（中等风险）
7. **桩替换** ——替换外部依赖（最高风险）

**障碍决策表。** 每种障碍类型映射到特定的处理模式。大语言模型并不发明变换——它从一个有限的菜单中根据障碍类型选择：

| 障碍 | 处理方式 |
|------|---------|
| 只读作用域逃逸 | 标注为常量 |
| 配置作用域逃逸 | 提升为参数 |
| 共享可变作用域逃逸 | 提升为 in-out 参数，或分解 |
| 基础设施污染 | 削除，或内联后提升 |
| 文件 I/O | 分解：核心 + I/O 外壳 |
| MPI / 通信 | 分解：内核 + 通信外壳 |
| 非确定性 | 分解：确定性核心 + 环境外壳 |
| 接口中的不透明类型 | 展平为原语 |
| 外部依赖 | 桩替换 |

创造力在于将模式**应用**于具体代码——阅读上下文、理解变量生命周期、判断哪些调用属于算法——而不在于发现模式本身。

**自底向上收敛。** 智能体沿调用图自底向上工作。叶函数首先被挖掘。当一个被调用者被认证为纯函数后，所有调用它的函数都自动少了一个继承障碍。这一反馈回路是结构性的而非启发式的——障碍沿调用关系向上传播，因此解决方案也沿已验证的被调用者向上传播。

```
叶节点 → 已验证叶节点 → 中层解锁 → 已验证中层 → ...
```

过程收敛到一个不动点：在当前检查模型下，从代码库中可提取的纯计算的最大集合。在对 NEMO（3,804 个函数）的试点分析中，可测试覆盖率从 31.9% 上升到 55.6%。这是初步测量结果；方法论及其在不同代码库间的推广是后续工作的主题。

### 3.4 证据与学习

版本树中的每条边都是一个类型化的记录：

```
Edge = {
    kind:           "rewrite" | "decompose",
    strategy:       string,
    barrier:        BarrierClass,
    shared_state:   [{ entity, lifecycle, resolution }],
    inspect_result: bool,
    inspect_feedback: string | null
}
```

这一结构以三种方式服务于神经-符号循环：

**反馈。** 失败的边携带 `inspect_feedback`——残余障碍。大语言模型读取反馈并调整下一次方案。循环不是盲目重试，而是有方向的搜索。

**解决方案库。** 成功的边积累为可检索的库。当大语言模型遇到具有相似障碍的新函数时，它检索在相似配置下成功的方案。符号组件积累的证据使神经组件日益高效。

**审计。** 每个已认证的纯计算都有可追溯的来源：从缠结的原始代码到已验证的局部化形式的变换链。证据是机器可读的、内容寻址的、永久保存的。

### 3.5 涌现的分区

不动点将代码库划分为两个区域：

**函数式核心** ——纯计算，已局部化，已验证。这些是不变量。它们可以翻译、优化和差分测试。它们是软件的实质。

**命令式外壳** ——I/O 封装、通信编排、数据库适配器。薄、机械、低风险。它们将所有计算委托给已验证的核心。它们是软件与世界的边界——类型化为状态接口。

这一分区不是强加的，而是**发现的**。智能体尝试每一个函数，应用每一个适用的模式，接受核心所认证的一切。分区从代码本身涌现——代码库的自然结构，通过系统地消解偶然缠结而显现。

### 3.6 这为何有效

神经-符号架构之所以有效，是因为两个组件具有互补的失败模式。

大语言模型的失败方式是**幻觉**——生成看似正确但包含微妙错误的代码：变量被默默重命名、边界条件被遗漏、副作用被引入。这些失败对大语言模型自身不可见。但它们对核心可见：幻觉产生的副作用在 `inspect` 中显示为障碍；遗漏的变量在 `drivergen` 中显示为编译失败；重命名的变量在 `run` 中显示为输出分歧。符号流水线捕获神经组件所遗漏的。

核心的失败方式是**不完备**——拒绝实际上是纯粹的代码，因为其纯粹性依赖于超出局部分析范围的不变量。一个读取模块变量的函数，如果该变量在整个代码库中恰好都是常量，则实际上是纯的，但在核心的局部扫描中却是不纯的。大语言模型解决这一问题：它阅读代码库，判定该变量是常量，然后将其提升为参数。神经组件解决符号组件看不到的。

组合覆盖了两种失败模式。大语言模型提出方案，核心检查。核心报告残余障碍，大语言模型调整。循环持续直到核心认证或搜索穷尽。全过程中，两个组件互不信任。信任在于协议：inspect 通过意味着可测试，可测试意味着任何人在任何时间用任何输入都可以检查。

## 4. 影响：以数值模拟为试验场

### 4.1 语料库

我们组建了一个包含约 110 个 Fortran 数值模拟项目的语料库——气候模型、海洋模型、大气物理模块、海冰动力学、水文代码、辐射传输方案、陆面模型。这些代码库跨越数十年开发历程，包含数百万行代码，代表了地球系统科学计算基础设施的相当一部分。（语料库仍在积极整理中；项目遴选、规模分布和预处理方法将另文报告。）

在此规模上应用三元分解，所揭示的不仅仅是单个代码库的结构，而是跨项目的共享结构。相同的计算反复出现——平流格式、扩散算子、状态方程求值、湍流闭合、辐射传输——由不同团队独立实现，风格各异，缠结方式不同，但所计算的不变量相同。从 NEMO 海洋热力学中提取的纯函数和从 MOM6 海洋热力学中提取的纯函数，是密切相关的物理关系的候选实现。一旦局部化并验证，它们便可以直接比较——其一致或不一致成为具体的、可重现的测量，而非需要人工逐行审阅的问题。

### 4.2 数值物理的本体论

所挖掘的纯函数自然地组织为一个本体论层次。最底层是原语：状态方程求值、通量计算、梯度算子、插值模板。其上是参数化方案：近似次网格过程的方案——对流、云微物理、边界层湍流、短波与长波辐射。再上是动力核心：在旋转球面上求解流体运动控制微分方程的求解器。

这一本体论并非由分类法强加。我们的假说是它可以被挖掘器**发现**——正如函数式核心/命令式外壳的分区在单个代码库内被发现一样。当多个代码库被挖掘时，反复出现的纯函数应按其所计算的物理而非所属的项目聚类。本体论将从不变量自身涌现：从微分方程出发，经由数值离散化，下至未分辨物理的参数化。在大规模上验证这一假说是本研究计划的核心目标。

### 4.3 跨模型比较

一旦纯函数被提取、验证并按本体论组织，一种新的科学分析便成为可能：在单个计算层面进行系统的跨模型比较。

当今的气候模型比较计划（CMIP）在整体模型输出层面运作——全球温度预测、降水格局、海平面估计。当模型出现分歧时，分歧的来源埋藏在数百万行缠结的代码中。识别是哪个参数化方案、哪种数值格式、哪个实现选择导致了分歧，需要领域专家进行艰苦的人工分析。

在三元分解下，比较变为机械化操作。两个模型的状态方程函数都是纯的。给它们相同的输入，比较输出。若有分歧：分歧被定位到该函数，并有具体的输入来证实。不一致性不再隐藏在黑箱之中——它被暴露为两个已验证纯计算之间的具体、可重现的差异。

这可以在整个本体论上规模化。每一个参数化方案、每一种数值格式、每一个通量计算——在 110 个项目间两两比较，自动进行，附带证据。科学价值不在于比较本身，而在于它所揭示的：哪些建模选择真正重要，真正的不确定性在哪里，哪些表面上的分歧只是实现上的差异。

### 4.4 加速与更精细分辨率

挖掘器所提取的纯函数是可移植的。它们可独立编译，具有显式签名，经过差分测试。这极大地扩展了经验证的翻译空间：计算核心可以翻译为 GPU 核函数、TPU 操作或其他加速器后端，并用同样的差分测试协议检验正确性。目标平台相关的约束（内存模型、精度语义、不支持的操作）依然存在，但分解将翻译问题从整个程序缩减到单个函数。

一个因其 Fortran 代码无法高效并行化而只能在 CPU 上以 100 公里分辨率运行的气候模型，在分解之后，其计算核心可以翻译为 JAX 或 CUDA，与原始版本验证一致，然后部署到加速器上。状态接口（MPI 通信、文件 I/O）作为薄外壳保留；纯函数——真正进行计算之所在——迁移到速度快数个数量级的硬件上。

更快的计算意味着更精细的分辨率。更精细的分辨率意味着模型能够解析先前被参数化处理的物理过程——海洋涡旋、对流单体、云结构。曾经作为次网格物理近似的参数化方案，变为对已解析物理的显式计算。科学上的收益是直接的：从前被假设的现象变成被计算的现象。

### 4.5 可微性与神经网络集成

三元分解所开启的最具深远意义的下游机会，或许是可微性。

一个拥有显式签名和经验证的输入输出行为的纯函数，不仅可以翻译为更快的实现，还可以翻译为**可微的**实现。一个计算对流参数化的 Fortran do 循环可以翻译为 JAX，通过差分测试验证，然后对其输入进行自动微分。

这将模型打开给基于梯度的方法。一个曾经是手工调参公式的参数化方案变成了一个可微函数，其参数可以从观测数据中学习。在本体论识别出具有清晰签名的自包含参数化方案之处，神经网络可以被插入——替换或增强该参数化——并通过周围的可微物理进行训练。并非每一个参数化方案都适合这种处理（有些涉及不连续性、刚性动力学或抵抗自动微分的隐式求解器），但分解能识别出候选方案并提供接口契约。

这种集成不是临时的。三元模型定义了接口：神经组件是一个与其所替换的参数化方案具有相同签名的纯函数。它通过同一协议验证——对原始版本和观测数据进行差分测试。它被插入到结构图中的同一位置。本体论告诉你**何处**可以插入神经模型。验证协议告诉你插入**是否正确**。可微实现告诉你**如何训练**它们。

长远愿景是一种计算基础设施，其中基于物理的模型与数据驱动的模型共存于同一形式表示中，由同一结构图连接，由同一协议验证，并可通过梯度联合优化。三元分解是这一愿景的必要基础：它将被计算的内容（纯函数——可微翻译的候选对象）与属于环境的内容（状态接口——不变）和组织方式（结构——神经组件嵌入的图）分离开来。全面实现这一愿景还需要在数值代码的自动微分、不连续性处理以及在长计算图上的可扩展训练等方面取得进展——这些都是活跃的研究前沿。

### 4.6 小结

本系统从数值模拟代码语料库中产出三项成果：
